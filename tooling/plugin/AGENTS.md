# @styleframe/plugin

The unplugin build integration. At build time it loads `styleframe.config.ts` plus every `*.styleframe.ts` file into one shared `Styleframe` instance, then serves the result as two virtual modules: `virtual:styleframe` (compiled TypeScript) and `virtual:styleframe.css` (all CSS). Published to npm, but app code normally reaches it through the barrel — `import styleframe from "styleframe/plugin/vite"` — per the root import rule.

## The two-faced virtual module

`resolveId` in [`src/plugin/index.ts`](./src/plugin/index.ts) inspects the **importer** of `virtual:styleframe`:

- Importer matches `\.styleframe\.ts$` → **extension face**: a factory returning the global instance, so the file can add tokens/recipes/utilities.
- Any other importer → **consumer face**: TypeScript generated by `@styleframe/transpiler` (`type: "ts"`) with recipe functions and selector constants.
- `virtual:styleframe.css` → all compiled CSS (`type: "css"`).

Initialization happens in `buildStart`: create a persistent jiti instance → load the config → discover `*.styleframe.ts` (sorted `alphabetical` or `depth-first`) → load them all → build an importree dependency graph → run the content scanner (if `scanner.content` is set) → register/tree-shake recipe utilities → register watch files → generate `.d.ts` into `.styleframe/`.

## HMR (Vite only)

`configureServer` + `handleHotUpdate` live under the `vite:` key — other bundlers get resolve/load but no selective invalidation. On change, `getFilesToInvalidate` walks the importree graph so only the changed file and its dependents are re-evaluated:

| Change | Action |
| --- | --- |
| `styleframe.config.ts` | Reload everything, invalidate all three virtual modules |
| A `*.styleframe.ts` file | Selective reload, invalidate CSS + consumer modules |
| Content file | Incremental scan; new utilities/recipe imports update CSS (and consumer) only |
| Tracked deep dependency (e.g. shared composable) | Selective reload of affected files |
| `*.styleframe.ts` added/removed | Load/drop it, rebuild the graph, re-scan, full-reload |

## Build-mode behavior

Two things happen only when `process.argv` includes `build`:

- **Recipe tree-shaking** (`recipes.treeshake`, default = build): the scanner scans `virtual:styleframe` imports in content files and drops recipes never imported by name. Namespace/dynamic imports, or zero detected imports, fall back to including everything with a warning. `recipes.include` force-keeps names.
- **Class-name minification** (`minify`, default = build): `generateGlobalCSS` builds `state.classNameLookup`, and a Vite `transform` hook ([`src/plugin/transform.ts`](./src/plugin/transform.ts)) rewrites matching literal utility class names in app source.

`generateGlobalCSS` also runs `validateInstanceLicense` from `@styleframe/license` on every CSS generation.

Options (`entry`, `include`/`exclude`, `loadOrder`, `dts`, `resolve.alias`, `scanner`, `recipes`, `minify`, `transpiler`, `silent`) are documented inline in [`src/plugin/types.ts`](./src/plugin/types.ts) — keep that file's JSDoc current, it is the option reference.

## Build & test

```bash
pnpm build             # tsdown (one dist entry per src/*.ts)
pnpm dev               # tsdown -w
pnpm test              # vitest run — tests in test/, not colocated
pnpm playground:dev    # run the Vue playground against the source plugin
```

Unit tests in [`test/`](./test/) cover the scanner and the generate/ modules. Cross-browser end-to-end coverage of virtual modules, HMR, themes, and tree-shaking lives in [`testing/integration/`](../../testing/integration/AGENTS.md) (Playwright, Vite fixture app) — add a spec there for anything observable in a real build.

## Pitfalls

- **All `*.styleframe.ts` files share one instance** loaded in order; two files exporting the same name throw `ExportCollisionError`, circular imports between them throw `CircularDependencyError` ([`src/plugin/errors.ts`](./src/plugin/errors.ts)). Shared code belongs in plain `.ts` composables (tracked as deep dependencies), not in another `.styleframe.ts`.
- **Dynamically constructed utility class names break minified builds.** `transformSourceClassNames` only rewrites literals matching the scanner pattern; a name built at runtime keeps its long form while the CSS ships shortened. Same story for recipes and tree-shaking: recipes must be imported by name from `virtual:styleframe` or they are dropped in build.
- **`@styleframe/plugin/types` is declared in `exports` but there is no `src/types.ts`** — `dist/types.js` is never built, so that subpath fails to resolve. The real types come from the root export.
- **The barrel (`styleframe/plugin/*`) exposes 8 adapters; this package exposes 9.** `engine/styleframe/src/plugin/` has no `bun.ts`, so Bun users must import `@styleframe/plugin/bun` directly.
