## Overview

The **Toast** is a transient notification element used for feedback messages, action confirmations, and brief status updates. The `useToastRecipe()` composable creates a fully configured [recipe](/docs/api/recipes) with color, variant, and size options &mdash; plus compound variants that handle the color-variant combinations automatically.

The companion `useToastBodyRecipe()` composable provides a content-area recipe that ensures the body fills available space and truncates long content correctly.

The Toast recipe integrates directly with the default [design tokens preset](/docs/design-tokens/presets) and generates type-safe utility classes at build time with zero runtime CSS.

## Why use the Toast recipe?

The Toast recipe helps you:

- **Ship faster with sensible defaults**: Get 9 colors, 4 visual styles, and 3 sizes out of the box with a single composable call.
- **Maintain consistency**: Compound variants ensure every color-variant combination follows the same design rules, including dark mode overrides.
- **Customize without forking**: Override base styles, default variants, or filter out options you don't need &mdash; all through the options API.
- **Stay type-safe**: Full TypeScript support means your editor catches invalid color, variant, or size values at compile time.
- **Integrate with your tokens**: Every value references the design tokens preset, so theme changes propagate automatically.
- **Compose flexible layouts**: The two-recipe pattern (container + body) supports icons, action buttons, and dismiss controls without imposing a rigid structure.

## Usage

::steps{level="4"}

#### Register the recipe

Add the Toast recipes to a local Styleframe instance. The global `styleframe.config.ts` provides design tokens and utilities, while the component-level file registers the recipes themselves:

:::code-tree{default-value="src/components/toast.styleframe.ts"}

```ts [src/components/toast.styleframe.ts]
import { styleframe } from 'virtual:styleframe';
import { useToastRecipe, useToastBodyRecipe, useButtonRecipe } from '@styleframe/theme';

const s = styleframe();

const toast = useToastRecipe(s);
const toastBody = useToastBodyRecipe(s);
const button = useButtonRecipe(s);

export default s;
```

```ts [styleframe.config.ts]
import { styleframe } from 'styleframe';
import { useDesignTokensPreset, useUtilitiesPreset } from '@styleframe/theme';

const s = styleframe();

useDesignTokensPreset(s);
useUtilitiesPreset(s);

export default s;
```

:::

#### Build the component

Import the `toast` and `toastBody` runtime functions from the virtual module and pass variant props to compute class names:

:::tabs
::::tabs-item{icon="i-devicon-react" label="React"}

```ts [src/components/Toast.tsx]
import { toast, toastBody, button } from "virtual:styleframe";

interface ToastProps {
	color?: "primary" | "secondary" | "success" | "info" | "warning" | "error" | "light" | "dark" | "neutral";
	variant?: "solid" | "outline" | "soft" | "subtle";
	size?: "sm" | "md" | "lg";
	title?: string;
	description?: string;
	icon?: React.ReactNode;
	action?: { label: string; onClick: () => void };
	dismissible?: boolean;
	onDismiss?: () => void;
	children?: React.ReactNode;
}

export function Toast({
	color = "neutral",
	variant = "solid",
	size = "md",
	title,
	description,
	icon,
	action,
	dismissible = false,
	onDismiss,
	children,
}: ToastProps) {
	const classes = toast({ color, variant, size });

	return (
		<div className={classes} role="status">
			{icon && <span>{icon}</span>}
			<div className={toastBody()}>
				{title && <strong>{title}</strong>}
				{description && <p>{description}</p>}
				{children}
				{action && (
					<button className={button({ variant: "link", size: "sm" })} onClick={action.onClick}>
						{action.label}
					</button>
				)}
			</div>
			{dismissible && (
				<button className={button({ variant: "ghost", size: "sm" })} onClick={onDismiss} aria-label="Dismiss">
					&times;
				</button>
			)}
		</div>
	);
}
```

## Colors

The Toast recipe includes 9 color variants: the 6 semantic colors (`primary`, `secondary`, `success`, `info`, `warning`, `error`) plus 3 neutral-spectrum colors (`light`, `dark`, `neutral`). Each color is combined with every visual style variant through compound variants, so you get consistent, predictable styling across all combinations &mdash; including dark mode overrides.

The `neutral` color adapts automatically: it uses a light appearance in light mode and a dark appearance in dark mode, making it the safest default for general-purpose toasts.

## Accessibility

- **Use `role="status"` for non-urgent toasts.** This creates a live region announced at the next pause in speech, without interrupting the current task. For urgent error toasts that require immediate attention, use `role="alert"` instead.

```html
<!-- Non-urgent confirmation -->
<div role="status" class="...">Your changes have been saved.</div>

<!-- Urgent error -->
<div role="alert" class="...">Payment failed. Please try again.</div>
```

- **Don't rely on color alone.** Always include descriptive text and pair with an icon to reinforce severity ([WCAG 1.4.1](https://www.w3.org/WAI/WCAG21/Understanding/use-of-color.html)).

```html
<!-- Correct: icon + descriptive text + color -->
<div role="status" class="...">
    <span>&#9888;</span>
    <strong>Warning:</strong> Your session will expire in 5 minutes.
</div>

<!-- Avoid: color alone with vague text -->
<div class="...">Note</div>
```

- **Dismissible toasts.** Use a `<button>` with `aria-label="Dismiss"` for the close control. Ensure toasts remain visible long enough for users to read and act on them &mdash; [WCAG 2.2.1](https://www.w3.org/WAI/WCAG21/Understanding/timing-adjustable.html) requires adjustable timing for auto-dismissing content.
- **Action buttons.** Use clear, specific labels like "Undo" or "View" rather than generic labels like "Click here". Ensure actions are keyboard accessible and focusable.
- **Verify contrast ratios.** The `solid` variant places light text on a colored background. Default tokens meet WCAG AA 4.5:1 contrast. If you override colors, verify with the [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/).

::tip
**Good practice:** When multiple toast types appear in your application, establish a severity hierarchy using variants. Use `solid` for critical errors, `soft` or `subtle` for informational messages, and `outline` for low-priority updates.
::

## Customization

### Overriding Defaults

The `useToastRecipe()` composable accepts an optional second argument to override any part of the recipe configuration. Overrides are deep-merged with the defaults, so you only need to specify the properties you want to change:

```ts [src/components/toast.styleframe.ts]
import { styleframe } from 'virtual:styleframe';
import { useToastRecipe } from '@styleframe/theme';

const s = styleframe();

const toast = useToastRecipe(s, {
    base: {
        borderRadius: '@border-radius.lg',
    },
    defaultVariants: {
        color: 'info',
        variant: 'soft',
        size: 'md',
    },
});

export default s;
```

### Filtering Variants

If you only need a subset of the available variants, use the `filter` option to limit which values are generated. This reduces the output CSS and keeps your component API focused:

```ts [src/components/toast.styleframe.ts]
import { styleframe } from 'virtual:styleframe';
import { useToastRecipe } from '@styleframe/theme';

const s = styleframe();

// Only generate success and error colors, with soft and subtle styles
const toast = useToastRecipe(s, {
    filter: {
        color: ['success', 'error'],
        variant: ['soft', 'subtle'],
    },
});

export default s;
```

::note
**Good to know:** Filtering also removes compound variants and adjusts default variants that reference filtered-out values, so your recipe stays consistent.
::

## API Reference

### `useToastRecipe(s, options?)`

Creates a full toast recipe with all variants and compound variant styling.

**Parameters:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `s` | `Styleframe` | The Styleframe instance |
| `options` | `DeepPartial<RecipeConfig>` | Optional overrides for the recipe configuration |
| `options.base` | `VariantDeclarationsBlock` | Custom base styles for the toast |
| `options.variants` | `Variants` | Custom variant definitions for the recipe |
| `options.defaultVariants` | `Record<keyof Variants, string>` | Default variant values for the recipe |
| `options.compoundVariants` | `CompoundVariant[]` | Custom compound variant definitions for the recipe |
| `options.filter` | `Record<string, string[]>` | Limit which variant values are generated |

**Variants:**

| Variant | Options | Default |
|---------|---------|---------|
| `color` | `primary`, `secondary`, `success`, `info`, `warning`, `error`, `light`, `dark`, `neutral` | `neutral` |
| `variant` | `solid`, `outline`, `soft`, `subtle` | `solid` |
| `size` | `sm`, `md`, `lg` | `md` |

### `useToastBodyRecipe(s, options?)`

Creates a recipe for the toast content area. Applies `flex: 1` and `min-width: 0` so the body fills the available space and truncates long content correctly.

**Parameters:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `s` | `Styleframe` | The Styleframe instance |
| `options` | `DeepPartial<RecipeConfig>` | Optional overrides for the recipe configuration |

## Best Practices

- **Choose colors by meaning, not appearance**: Use `success` for confirmations, `error` for failures, and `warning` for caution messages &mdash; this keeps your UI consistent when tokens change.
- **Use `neutral` for generic notifications**: The neutral color adapts to light and dark mode automatically, making it the safest default for general-purpose toasts.
- **Prefer `subtle` or `soft` for informational toasts**: Reserve `solid` for critical notifications that demand immediate attention. Too many solid toasts create visual fatigue.
- **Pair an icon with the message**: Icons reinforce the toast's severity and improve scannability, especially for users who cannot distinguish colors.
- **Include an action when reversible**: If the action can be undone (e.g., "Undo delete"), add an action button so users can recover quickly.
- **Keep toast messages short**: Toasts are transient &mdash; use concise titles and descriptions that can be read at a glance.
- **Filter what you don't need**: If your application only uses a few colors, pass a `filter` option to reduce generated CSS.
- **Override defaults at the recipe level**: Set your most common variant combination as `defaultVariants` so component consumers write less code.

## FAQ

::accordion

:::accordion-item{label="How do compound variants work in the Toast recipe?" icon="i-lucide-circle-help"}
The Toast recipe uses compound variants to map each color-variant combination to specific styles. For the 6 semantic colors (`primary`, `secondary`, `success`, `info`, `warning`, `error`), styles are generated programmatically using a `flatMap` over the color array. For the `light`, `dark`, and `neutral` colors, compound variants are defined individually with explicit dark mode overrides. This approach keeps the individual `color` and `variant` definitions clean while handling all 36 combinations (9 colors &times; 4 variants) automatically.

[Learn more about compound variants &rarr;](/docs/api/recipes#compound-variants)
:::

:::accordion-item{label="What is the difference between light, dark, and neutral colors?" icon="i-lucide-circle-help"}
`light` always uses white and gray-100 backgrounds regardless of the color scheme &mdash; it stays light even in dark mode. `dark` always uses gray-800 and gray-900 backgrounds &mdash; it stays dark even in light mode. `neutral` adapts to the current color scheme: it appears light in light mode and dark in dark mode. Use `neutral` when you want the toast to blend naturally with the surrounding interface.
:::

:::accordion-item{label="Can I add custom colors beyond the default nine?" icon="i-lucide-circle-help"}
Yes. Override the `variants.color` and `compoundVariants` options to add new colors. You'll need to define the empty color variant and add compound variants for each visual style:

```ts [src/components/toast.styleframe.ts]
const toast = useToastRecipe(s, {
    variants: {
        color: {
            brand: {},
        },
    },
    compoundVariants: [
        { match: { color: 'brand', variant: 'solid' }, css: { background: '@color.brand', color: '@color.white' } },
        { match: { color: 'brand', variant: 'outline' }, css: { color: '@color.brand', borderColor: '@color.brand' } },
        // ... soft and subtle
    ],
});
```
:::

:::accordion-item{label="What's the difference between soft and subtle variants?" icon="i-lucide-circle-help"}
Both use a light tinted background with colored text. The difference is that `subtle` also adds a matching colored border, giving the toast more visual definition. Use `soft` when you want a gentler appearance, and `subtle` when the toast needs slightly more structure.
:::

:::accordion-item{label="What is the toast body recipe for?" icon="i-lucide-circle-help"}
The `useToastBodyRecipe()` creates a minimal recipe (`flex: 1`, `min-width: 0`) for the content area between the icon and dismiss button. The `flex: 1` ensures the body fills all available space, and `min-width: 0` prevents long text from overflowing the toast container. Without the body recipe, text-heavy toasts can push the dismiss button out of view.
:::

:::accordion-item{label="How do I handle toast positioning and animation?" icon="i-lucide-circle-help"}
The Toast recipe handles only the visual styling of individual toast elements. Positioning (e.g., top-right, bottom-center), stacking, and animation are the responsibility of your toast container/manager component. Use CSS `position: fixed` with appropriate offsets for the container, and CSS transitions or an animation library for enter/exit animations.
:::

:::accordion-item{label="What's the difference between a Toast and a Callout?" icon="i-lucide-circle-help"}
Both share similar visual styling (colors, variants, compound variants), but they serve different purposes. **Toasts** are transient notifications that appear temporarily, often include action buttons (like "Undo"), and are positioned outside the normal document flow. **Callouts** are persistent inline elements embedded in the page content with support for horizontal and vertical orientations. Use a toast for ephemeral feedback and a callout for persistent contextual information.
:::

:::accordion-item{label="How does filtering affect compound variants?" icon="i-lucide-circle-help"}
When you use the `filter` option, compound variants that reference filtered-out values are automatically removed. For example, if you filter `variant` to only `['solid', 'outline']`, all compound variants matching `soft` or `subtle` are excluded from the generated output. Default variants are also adjusted if they reference a removed value.
:::

:::accordion-item{label="Can I use the Toast recipe without the design tokens preset?" icon="i-lucide-circle-help"}
The Toast recipe references design tokens like `@color.primary`, `@font-size.sm`, and `@border-radius.md` through string refs. These tokens need to be defined in your Styleframe instance for the recipe to generate valid CSS. The easiest way is to use `useDesignTokensPreset(s)`, but you can also define the required tokens manually.
:::
